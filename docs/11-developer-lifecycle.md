# 11. The Aura-NX Developer Lifecycle

This document provides a comprehensive end-to-end walkthrough of the developer experience (DX) when building Nintendo Switch applications using the Aura-NX ecosystem. It is designed to guide both human engineers and automated systems through a professional, high-velocity workflow.

---

## Phase 1: Environment Initialization (The First Hour)

Before writing code, the studio establishes the "Aura-NX Command Center."

1.  **Sysmodule Deployment**:
    *   The engineer copies `AuraNX.nsp` to the Switch SD card (`/atmosphere/contents/`).
    *   The console is rebooted. The sysmodule immediately begins broadcasting `AURA_ANNOUNCE` beacons on the LAN.
2.  **Server Initialization**:
    *   The developer starts the Aura-NX server on their workstation.
    *   The server automatically "claims" the console via UDP discovery, establishing a JSON-RPC link for telemetry and a TCP link for the Asset Server.
3.  **Project Integration**:
    *   The developer links their application project against `libaura-nx.a`.
    *   In the application's main entry point, they call `auraNxInit("192.168.1.15")` and add `auraNxUpdate()` to the main loop.

---

## Phase 2: High-Velocity Development (The Daily Loop)

Aura-NX eliminates the "SD card shuffle," allowing for a near-instant feedback loop.

1.  **Network Asset Streaming**:
    *   Instead of loading assets from the SD card, the engine uses `fopen("net:/textures/skybox.png")`.
    *   The call is intercepted by the `devoptab` layer, which fetches the file directly from the PC's project folder over the high-speed TCP bridge.
2.  **Instant Asset Hot-Reload**:
    *   An artist or developer modifies a fragment shader on the PC.
    *   The PC-side file watcher detects the change and triggers a reload signal.
    *   The Switch receives a UDP `RELOAD` signal, invokes the application's registered callback, and swaps the shader in GPU memory **mid-frame**.
    *   **Result**: 100ms latency from "File Save" to "Visual Confirmation" on target hardware.

---

## Phase 3: Automated Debugging & Control

Aura-NX provides the granular control required for autonomous development and testing.

1.  **Remote Execution Control**:
    *   When a logic bug occurs, the developer uses the `set_breakpoint` tool.
    *   The Atmosphere GDB stub freezes the application. The server symbolicates the instruction pointer using the stored `.elf` file.
2.  **Input Injection for Unit Testing**:
    *   To verify a complex UI flow, a script or agent issues a sequence of `send_input` commands (e.g., `A`, `Down`, `A`).
    *   The console executes these inputs as if a physical controller were connected, allowing for 100% automated regression testing of application mechanics.
3.  **Visual QA via Screen Capture**:
    *   The system calls `capture_screen`. The sysmodule grabs the framebuffer, compresses it, and sends it back as a JPEG.
    *   Computer vision models analyze the JPEG to ensure UI elements are correctly aligned and no rendering artifacts are present.

---

## Phase 4: Profiling & Production-Grade QA

Moving from "it works" to production-ready quality.

1.  **Live Telemetry Dashboard**:
    *   The Lead Programmer opens the **Aura Target Manager** in their browser.
    *   They monitor real-time graphs of **GPU Load** and **CPU Clocks** while running stress tests.
2.  **Self-Healing Crash Analysis**:
    *   If the application crashes, the sysmodule immediately detects the new report in `/atmosphere/crash_reports/`.
    *   The dump is streamed to the PC. The server symbolicates the stack trace and posts a detailed bug report, including the exact line of code and a hardware state snapshot.
3.  **HIL (Hardware-in-the-Loop) Verification**:
    *   The `verify_all.sh` script runs in the studio's CI pipeline.
    *   It verifies that the current build maintains the 16MB memory ceiling and that the telemetry latency remains within defined performance thresholds.

---

## Phase 5: Release & Post-Mortem

Finalizing the "Gold" master.

1.  **Binary Sign-off**:
    *   The studio lead reviews the `docs/10-audit-signoff.md` generated by the automated audit swarm.
2.  **Symbol Archival**:
    *   The final `.elf` and `.map` files are archived. This ensures that any crash reports received from field testers can be symbolicatied against the exact release build.
3.  **Fleet Decommissioning**:
    *   Aura-NX is disabled via a compiler flag for the retail build, ensuring no dev-tool overhead or security hooks remain in the final consumer product.

---

**Aura-NX transforms Switch development from a manual, hardware-constrained struggle into a high-performance, professional engineering pipeline.**
